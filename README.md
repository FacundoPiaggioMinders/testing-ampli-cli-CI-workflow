# testing-ampli-cli-CI-workflow

A small Next.js (App Router, TypeScript, Tailwind v4) sandbox for exercising the Amplitude
[Ampli CLI](https://www.docs.developers.amplitude.com/data/ampli/) and validating it inside a CI
workflow.

The app initializes `@amplitude/analytics-browser` from a `NEXT_PUBLIC_AMPLITUDE_API_KEY` and
exposes a handful of test buttons that fire events you can verify in Amplitude's
**User Lookup → Stream**.

## Stack

- Next.js 16 (App Router) on React 19
- TypeScript, Tailwind v4
- `@amplitude/analytics-browser` (runtime SDK)
- `@amplitude/ampli` (dev dependency, drives the CLI)

## Project layout

```
.github/workflows/
  ampli.yml                   # PR CI: `ampli status --update` against the plan
src/
  app/
    layout.tsx                # Mounts <AmplitudeProvider>
    page.tsx                  # Test event buttons + recent-events log
  components/
    AmplitudeProvider.tsx     # Initializes the SDK on mount (client component)
  lib/
    analytics.ts              # Thin wrapper around amplitude-browser
  ampli/                      # Generated by `ampli pull` (gitkept until first pull)
ampli.json                    # Created by `ampli pull` on first run
```

## First-time setup

```bash
npm install
cp .env.local.example .env.local
# paste your Amplitude project API key into .env.local
npm run dev
```

Open http://localhost:3000 and click the test buttons. Without an API key the app still works —
events are appended to the on-screen log but not delivered to Amplitude.

## Ampli CLI workflow

The CLI is wired up via `@amplitude/ampli` as a dev dependency. Use it through `npx` or the
provided npm scripts.

```bash
# one-time auth
npx ampli login

# pull the tracking plan and (re)generate src/ampli/
npm run ampli:pull

# diff local generated code against the latest plan
npm run ampli:status

# push local plan changes back to Amplitude (advanced — usually CI-only)
npm run ampli:push
```

The first `ampli pull` walks you through selecting org → workspace → source → branch and writes
an `ampli.json` at the repo root. Generated TS lives in `./src/ampli`. After the first pull, swap
calls in `src/app/page.tsx` from the generic wrapper to the typed Ampli API:

```ts
// before (works without ampli)
import { track } from "@/lib/analytics";
track("Button Clicked", { label: "primary" });

// after `ampli pull`
import { ampli } from "@/ampli";
ampli.buttonClicked({ label: "primary" });
```

## CI workflow

`.github/workflows/ampli.yml` runs on every PR against `main`, using the official
`amplitudeinc/ampli-all` Docker image (ampli pre-installed) and runs:

```
ampli status -t $AMPLITUDE_API_TOKEN --update
```

`--update` patches the **Implementation Status** column in Amplitude Data based on what the
branch's code actually references — useful for PR review. Drop the flag if you want a read-only
check.

### Required setup

1. Generate a personal API token in Amplitude:
   `Settings → Account → API Token`.
2. Add it to the repo: `Settings → Secrets and variables → Actions → New repository secret`,
   name **`AMPLITUDE_API_TOKEN`**.
3. Commit `ampli.json` after your first local `ampli pull` so the CI container can resolve the
   right org / workspace / source.

### Optional extra guards

- `ampli pull --omit-api-keys` + `git diff --exit-code src/ampli` to fail when the committed
  generated code drifts from the plan.
- `next build` to catch type errors introduced by a regenerated `src/ampli/`.
